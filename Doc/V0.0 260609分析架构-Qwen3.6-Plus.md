## 项目架构概述

这是一个基于 **ESP-IDF** 的 IoT 网关项目，运行在 ESP32-C3 平台上，使用 FreeRTOS 多任务架构：

```
app_main()
├── sensor_data_mgr_init()          # 传感器数据管理器（共享数据中心）
├── task_monitor (P1)               # 任务监控 - LED指示空闲状态
├── task_app_wifi_manage (P4)       # WiFi管理
├── task_app_mqtt (P3)              # MQTT客户端
├── task_app_mainbusiness (P3)      # 主业务 - 状态机
├── task_app_sensor_comm (P10!)     # BMP280传感器
├── dht11_task (P23)                # DHT11温湿度传感器
├── mics5524_task (P5)              # 气体传感器
└── task_button (P5)                # 按键处理
```

**核心模块：**
- `blink_example_main.c` — 入口文件 + DHT11/MICS5524 直接实现
- `sensor_data_mgr.c/h` — 共享传感器数据中心（带互斥锁）
- `task_app_mainbusiness.c/h` — 状态机驱动的主业务逻辑
- `task_app_mqtt.c/h` — MQTT 连接与数据上报
- `task_app_wifi_manage.c/h` — WiFi 连接管理
- `task_app_sensor_comm.c/h` — BMP280 I2C 传感器通信

## 问题清单（按优先级排列）

### P0 — 严重 Bug

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| 1 | **任务优先级不一致** | `task_app_sensor_comm.c:129` | 注释写 `Normal priority(3)`，实际传的是 `10`，与预期不符 |
| 2 | **MQTT 竞态条件** | `task_app_mqtt.c:65` → `task_app_mainbusiness.c:65` | MQTT 收到消息后调用 `app_send_command()`，但 command queue 在 `handle_state_init()` 中创建。如果 MQTT 在 INIT 状态完成前收到消息，发送会失败 |
| 3 | **DHT11 禁用所有中断** | `blink_example_main.c:66-104` | `portDISABLE_INTERRUPTS()` 禁用所有中断，期间 FreeRTOS tick 中断也被禁用，系统时钟会"冻结"。如果 DHT11 响应慢，整个系统挂起 |
| 4 | **互斥锁安全隐患** | `sensor_data_mgr.c:20-32` | `lock_data()` / `unlock_data()` 是分开调用的。如果某个 update 函数在持有锁期间被意外删除（`vTaskDelete`），锁永远不会释放，所有后续访问死锁 |

### P1 — 高优先级

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| 5 | **WiFi 密码硬编码** | `task_app_wifi_manage.c:13-14` | SSID 和密码直接写死在代码中，安全隐患，不便携 |
| 6 | **MQTT 发布无超时控制** | `task_app_mqtt.c:126-151` | 无论数据是否变化，每 3 秒强制发布一次。在网络差时可能造成消息堆积 |
| 7 | **BMP280 读取无错误恢复** | `task_app_sensor_comm.c:95-118` | 读取失败后只是打印错误日志，没有任何恢复机制（重试、重新初始化等） |

### P2 — 中优先级

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| 8 | **I2C 配置重复** | `task_app_sensor_comm.c:67-73` + `scan_i2c_bus()` 内 | I2C GPIO 配置在多处重复定义（GPIO6/7），修改时需要同步多处 |
| 9 | **引脚定义分散** | 多个文件中 | GPIO 引脚定义分布在 `blink_example_main.c` 和 README 中，没有在统一头文件中管理 |
| 10 | **DHT11 5s vs README 2s** | `blink_example_main.c:154` vs `README.md` | README 说 "每2秒更新一次"，实际代码是 5 秒延迟 |
| 11 | **状态机过于简单** | `task_app_mainbusiness.c` | 当前状态机仅控制 LED ON/OFF，实际业务价值低，没有充分发挥状态机的设计模式 |
| 12 | **无集中配置管理** | 全局 | MQTT broker、topic、WiFi 凭证、传感器采样间隔等全部散落在各处 |

### P3 — 低优先级

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| 13 | **任务栈空间可能过大** | `app_main.c` | WiFi(8192)、MQTT(8192) 等任务栈分配较大，ESP32-C3 RAM 有限 |
| 14 | **JSON 拼接方式脆弱** | `sensor_data_mgr.c:92-113` | 使用 `snprintf` 手动拼接 JSON，无转义处理，buffer 大小硬编码 |
| 15 | **LED 状态管理混乱** | `blink_example_main.c` | `s_led_state` 在 `task_button` 和 `task_app_mainbusiness` 中各有一份，职责不清晰 |
| 16 | **`scan_i2c_bus` 临时对象配置** | `task_app_sensor_comm.c:24-37` | 每次扫描循环都创建新的 `i2c_dev_t`，效率低 |
